# HannsDB V1 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build HannsDB as a local Rust agent database with an embedded core, a minimal daemon surface, and a Python-first `VectorDBBench` integration path using `knowhere-rs` HNSW underneath. The project also has a parallel validation goal: HannsDB development must continuously verify that `knowhere-rs` is stable, reusable, and semantically compatible as the ANN engine for the target path.

**Architecture:** HannsDB owns catalog, storage, recovery, and query orchestration. `knowhere-rs` is wrapped behind a narrow HNSW adapter. Python binding and daemon both sit on top of the same core, while `VectorDBBench` integration is implemented against the Python binding first.

**Tech Stack:** Rust workspace, `knowhere-rs`, PyO3, HTTP or Unix socket server crate, Python 3.11, VectorDBBench

## Execution Notes

- Use `knowhere-rs` HNSW only in v1.
- Treat `knowhere-rs` verification as a first-class project goal, not a side effect:
  - every new HannsDB ANN milestone should include at least one focused `knowhere-rs`-backed verification step
  - record semantic mismatches or integration caveats as HannsDB/`knowhere-rs` compatibility notes, even when HannsDB remains the layer that must adapt
- Do not start IVF-PQ or DiskANN work before the benchmark smoke path is passing.
- Keep daemon implementation thin and non-authoritative.
- Split worker ownership by subsystem:
  - worker A: `hannsdb-core` + `hannsdb-index`
  - worker B: `hannsdb-py` + `VectorDBBench` integration
  - controller: daemon, reviews, integration, and verification
